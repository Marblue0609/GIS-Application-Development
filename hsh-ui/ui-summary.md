# hsh-ui 前端阶段改进说明

## 1. 当前模块状态

`hsh-ui` 是 CityTaste 项目的前端 UI 与地图交互模块，当前已从本地 GeoJSON 原型逐步接入 FastAPI 后端。页面主要包含首页与工作台两部分：

- 首页：项目入口、功能概览、当前数据状态展示。
- 工作台：餐厅检索、范围分析、路线预览、打卡清单、Cesium 地图交互。

当前前端可以在后端未启动时使用本地 GeoJSON 兜底，在后端启动时优先调用 API。

## 2. 本地运行与检查命令

进入项目根目录后运行：

```bash
npm run dev --prefix hsh-ui
```

浏览器打开：

```text
http://127.0.0.1:5173/
```

提交前建议检查：

```bash
npm run lint --prefix hsh-ui
npm run build --prefix hsh-ui
```

目前 `build` 会出现 Cesium 包体较大的 warning，这是 Cesium 前端项目中比较常见的体积提示，不是语法错误。

## 3. 上一阶段已有基础

上一阶段前端已经完成了基础页面框架和主要功能入口：

- Cesium 2D 地图展示。
- 餐厅、地标、交通点位显示。
- 餐厅搜索、菜系筛选、价格筛选、评分筛选。
- 随机推荐、结果列表、点位点击、地图飞行定位。
- 范围分析、路线预览、打卡清单等功能面板。
- 本地 GeoJSON 兜底数据加载。

## 4. 本次主要改进

### 4.1 UI 与页面提示优化

- 调整首页和工作台品牌区域的排版间距，使 `GIS FOOD ANALYSIS` 与 `CityTaste` 更紧凑。
- 去掉地图左上角过大的图层统计面板，只保留当前选中点提示。
- 删除路线面板中的开发提示文案，例如 `AMAP_KEY`、后端配置、兜底说明等，避免用户界面出现技术实现细节。
- 将部分提示语改为更面向用户的表达，例如：
  - `已加入打卡清单`
  - `已移出打卡清单`
  - `已生成路线`
  - `已为你推荐一家餐厅`

### 4.2 地图点位表现优化

- 将原来的普通圆点改为 SVG pin 图标，参考手机地图常见的圆形加小尖角样式。
- 点位使用更高分辨率 SVG 生成，减少图标放大后的模糊感。
- 保留鼠标悬停卡片，用于展示点位名称、类型和评分。
- 餐厅、地标、交通点位都增加了按视角缩放的分级显示：
  - 小比例尺下减少点位数量，优先显示高优先级点。
  - 中近景下增加点位数量，同时做屏幕空间避让。
  - 放大到足够近时显示更多点位。
- 当用户选择某个菜系分类时，餐厅点位不再分级裁剪，尽量完整显示当前筛选结果；交通点位仍继续分级显示。

### 4.3 路线交互优化

- 路线生成后，路线经过的清单点会使用路线高亮图标显示。
- 普通点选高亮和路线高亮做了状态区分，避免两种状态混在一起。
- 生成路线时会清除普通选中态，让地图焦点切换到路线结果。
- 新增“清除路线”按钮，可以手动退出路线态。
- 修改筛选条件时会自动清除已有路线，避免筛选结果和旧路线高亮互相覆盖。
- 路线绘制时会把清单中的 `waypoints` 贴回路线 `path`，让路线视觉上尽量经过清单餐厅点。

### 4.4 API 接入后的适配

- 后端接通后，餐厅数据优先来自 `/api/restaurants` 和 `/api/restaurants/search`。
- 搜索接口默认返回数量从 120 调整为 500，避免 API 接入后结果数量明显少于本地 GeoJSON。
- 后端返回的原始 `category` 保留用于展示和筛选，同时新增 `categoryGroup` 用于前端菜系归类和地图配色。
- 当前如果餐饮点统一颜色也可以接受，`categoryGroup` 仍可作为后续恢复分类配色的基础字段。

## 5. 当前仍需关注的问题

### 5.1 双标问题

部分餐厅在选中、随机推荐、搜索选中或路线交互后，仍可能出现“一大一小两个图标”的情况。当前已经做过的处理包括：

- 删除 overlay 中额外创建选中餐厅点的逻辑。
- 选中点不再新增实体，而是在原有 billboard entity 上更换图标和尺寸。
- 路线经过点也不新增一层点位，而是在原有餐厅实体上使用路线高亮样式。

如果后续仍然出现双标，建议优先检查数据中是否存在同名或近距离重复餐厅，其次再检查 Cesium datasource 是否有旧图层残留。这个问题目前不建议继续单纯通过加大前端坐标去重阈值解决，因为阈值过大会误伤商场内真实相邻餐厅。

### 5.2 路线和点位坐标对齐

前端目前假设后端返回的餐厅点和路线 `path` 都是 WGS84。若路线仍和餐厅点明显错位，建议后续和后端一起检查：

- `/api/restaurants` 的餐厅 `lng/lat`
- `/api/route/plan` 的 `waypoints`
- `/api/route/plan` 的 `path`
- PostGIS 中餐厅几何字段的坐标系
- 高德路线返回坐标是否已经正确从 GCJ-02 转回 WGS84

## 6. 后续建议

- 路线态、筛选态、普通选中态继续保持互斥，避免地图上同时出现多个视觉焦点。
- 如果后续需要“筛选时保留路线”，建议加一个明确的路线状态条，而不是让路线高亮静默覆盖筛选状态。
- 后端接口最好保持统一字段结构，便于前端统一 normalize：

```text
id
layerType
name
rating
phone
category
price
address
lng
lat
```

路线接口建议继续返回：

```text
method
totalDistanceM
waypoints
path
pathCoordinateSystem
```

其中 `waypoints` 最好直接复用餐厅标准字段，便于前端统一处理。
